APPUNTI DI SISTEMI OPERATIVI PER LE ESERCITAZIONI IN LABORATORIO

PARTE 6 -  SINCRONIZZAZIONE DI PROCESSI: I SEMAFORI XSI

*******************************************************

SEMAFORI

*******************************************************

In questa sezione vedremo i semafori disponibile in Unix come parte dell'estensione XSI, derivanti dalla versione System V. Tali semafori sono una versione generalizzata dei semafori proposti in letteratura, nel senso che con le chiamate di sistema per operare su di essi si possono:
  • riprodurre le operazioni classiche, originariamente chiamate P e V da Dijkstra, ridenominate su alcuni testi, fra cui quello di Tanenbaum, "down" e "up", oppure "wait" e "signal" (ovviamente in Unix non si possono chiamare così, visto che quei nomi sono già usati per altre system calls)
  • ma anche effettuare operazioni più generali, come vedremo.
***************
 CREAZIONE
***************

Un primo aspetto importante da tenere presente nell'uso dei semafori (come per le altre strutture di Inter Process Communication di XSI, per esempio i segmenti di memoria condivisa) è la natura persistente di tali strutture. Come per la memoria condivisa IPC,  proprio perché servono a più processi, non vengono automaticamente deallocate quando termina il processo che ne ha richiesto l'allocazione. E devono essere richieste al sistema operativo, nel caso dei semafori con la chiamata:

SEMGET:

     int semget(key_t key, int nsems, int semflg);

che restituisce l'identificatore di un vettore di "nsems" semafori associati alla chiave "key" (il terzo parametro "semflg" serve per diverse opzioni che vedremo in parte in seguito). L'identificatore restituito potrà poi essere utilizzato nelle operazioni (tipo down e up) illustrate nel seguito.

La chiave (analogamente a quanto accade nel caso della memoria condivisa), serve a fare in modo che i processi di un insieme A, che devono sincronizzarsi attraverso uno o più semafori, possano acquisire lo stesso semaforo, che verrà effettivamente condiviso, mentre un altro insieme B di processi, che devono sincronizzarsi tra di loro ma non con quelli in A, possa acquisire un semaforo che sia garantito essere diverso da quello che usano i processi dell'insieme A. La garanzia di ottenere un semaforo diverso serve ad evitare ad esempio che una "up" destinata a risvegliare un processo dell'insieme A abbia invece l'effetto di risvegliarne uno dell'insieme B.

Analogamente alla shmget, la semget può quindi essere usata da diversi processi che condividono una data chiave mykey con:

semid = semget(mykey, 1, 0600 | IPC_CREAT )

oppure da un processo genitore, passando i seguenti parametri:

semid = semget(IPC_PRIVATE, 1, 0600)

in questo caso tutti i processi discendenti potranno condividere fra loro e con il comune genitore il semaforo identificato da semid.

Il comando ipcs digitato da un terminale, permette di visualizzare tutti gli oggetti della categoria IPC presenti nel sistema.
Per eliminare da shell i semafori creati, non eliminati dai programmi che li hanno utilizzati, ma ritenuti non più utili, si può utilizzare il comando ipcrm -s id del semaforo .

********************************
INIZIALIZZAZIONE E RIMOZIONE
********************************

Varie operazioni su un semaforo (non quelle più significative di sincronizzazione) possono essere effettuate con:
SEMCTL:

    int semctl(int semid, int semnum, int cmd, ...);

che segue un'operazione specificata da cmd su semid, o in qualche caso sul suo elemento identificato da semnum.

Le operazioni più rilevanti sono:

cmd = GETVAL : restituisce il valore della componente semnum;

cmd = GETNCNT : restituisce il numero di processi in attesa (con semop, si veda più avanti) che il valore della componente semnum diventi più grande di quello corrente

cmd = SETVAL : assegna alla componente semnum il valore passato nel 4o argomento che è del tipo:

     union semun {
         int val;
         struct semid_ds *buf;
         ushort *array;
     }

mentre il valore da assegnare deve essere messo nel campo val. A seconda delle versioni e opzioni di installazione, questa "union semun" può dover essere dichiarata esplicitamente.

cmd = IPC_RMID rimuove il vettore di semafori semid

Le operazioni GETVAL e SETVAL rendono possibile usare i semafori come qualunque variabile, infatti permettono di leggerne il valore e modificarlo; tuttavia è raccomandato utilizzare SETVAL esclusivamente per inizializzare il valore del semaforo, in un momento in cui i processi che devono utilizzarlo per sincronizzarsi non hanno ancora iniziato la loro attività, e successivamente utilizzare soltanto la semop descritta più avanti (che permette di realizzare operazioni analoghe a down e up) per operare sugli stessi.

************************************************
OPERAZIONI DI SINCRONIZZAZIONE DOWN E UP
************************************************

Lo operazioni di sincronizzazione, incluse down e up, possono essere realizzate con la system call:

SEMOP:

     int semop(int semid, struct sembuf *sops, size_t nsops);

Il tipo struct sembuf include i seguenti campi: 
	unsigned short sem_num;  /* semaphore number */
	short sem_op;            /* semaphore operation */
	short sem_flg;           /* operation flags */
La system call semop esegue sul semaforo semid le nsops operazioni contenute nell'array sops.
Si può dichiarare un vettore:
struct sembuf sops[N];
riempirlo con la descrizione delle operazioni da fare (per i dettagli di come si descrivono, vedere più avanti), e chiamare:
semop(semid,sops,N);
Oppure, nel caso di una singola operazione (quello più comune) si può dichiarare:
struct sembuf sb;
e chiamare:
semop(semid,&sb,1);
Ogni elemento dell'array sops specifica una operazione, ogni operazione ha 3 campi:

sem_num: numero della componente del vettore di semafori su cui operare;
sem_op: operazione da fare (vedi sotto);
sem_flg: opzioni

Con valori diversi di sem_op, in particolare 1 e -1, si possono realizzare come si vedrà più avanti la up e la down, ma in generale il comportamento è il seguente:
  • se sem_op > 0, il valore del semaforo è incrementato di sem_op;
  • se sem_op < 0, allora :
    •   se |sem_op| <= valore semaforo, il valore del semaforo viene decrementato di |sem_op|;
    •   se |sem_op| > valore semaforo, allora se sem_flg & IPC_NOWAIT è vero si ritorna subito, altrimenti il processo viene sospeso fino a che si verifica una delle seguenti condizioni:
      • in conseguenza dell'aumento del valore del semaforo si ha |sem_op| <= valore semaforo , nel qual caso il valore del semaforo viene decrementato di |sem_op|;
      • il vettore di semafori semid viene rimosso (quindi attenzione a non rimuovere semafori che sono utilizzati da qualche processo, se non nel caso in cui ci si accorga che il normale lavoro dei processi che li usano per sincronizzarsi non può più proseguire);
      • il processo riceve un segnale da "catturare" (ha associato alla ricezione del segnale una funzione da eseguire);
  • se sem_op = 0, il processo viene sospeso in attesa che il valore del semaforo diventi zero (si esce subito se lo è già)
I seguenti sono esempi di realizzazione delle operazioni up e down "classiche" sui semafori mediante i semafori Unix (insieme ad una funzione da utilizzare per l'inizializzazione di un semaforo).
Per up e down si usano i valori 1 e -1 per sem_op.
Se il semaforo vale 0 e vi sono diversi processi sospesi in attesa di completare una down (operazione con sem_op=-1), viene risvegliato uno solo di essi; in questo modo il valore del semaforo non diventa mai negativo. Successive up risveglieranno gli altri. Può darsi che una specifica implementazione risvegli questi processi nell'ordine in cui sono stati sospesi (First In First Out), ma non vi è alcuna garanzia che lo stesso avvenga in una diversa versione di Unix.

semfun.c
::::::::::::::
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/sem.h>
#include <stdio.h>

int down(int semid, int semnum)	
	/* operazione DOWN sulla componente semnum di semid */
{
  int r;
  struct sembuf sb;

  sb.sem_num=semnum;
  sb.sem_op=-1;
  sb.sem_flg=0;
  r=semop(semid,&sb,1);
  if (r==-1) perror("semop in down");
  return r;
}

int up(int semid, int semnum)	
	 /* operazione UP sulla componente semnum di semid */
{
  int r;
  struct sembuf sb;

  sb.sem_num=semnum;
  sb.sem_op=1;
  sb.sem_flg=0;
  r=semop(semid,&sb,1);
  if (r==-1) perror("semop in up");
  return r;
}

int seminit(int semid, int semnum, int initval)
	/* inizializzazione con initval della componente semnum di semid */
{
  int r;
  union semun
  {
    int val;
    struct semid_ds *buf;
    unsigned short *array;
  } arg;
  arg.val=initval;
  r=semctl(semid,semnum,SETVAL,arg);
  if (r==-1) perror("semop in up");
  return r;
}

Nel seguente esempio un processo alloca e inizializza un semaforo, poi genera un processo figlio; i due processi usano il semaforo per sincronizzarsi, in particolare il padre aspetta una operazione UP del figlio.
::::::::::::::
sem.c
::::::::::::::
#include "semfun.h"

#include <stdio.h>
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/sem.h>

int main()
{
  int semid,val,n;

  semid = semget(IPC_PRIVATE,1,0600);
  if (semid == -1) perror("Creazione semaforo");
  printf("Identificatore semaforo: %d\n",semid);
  seminit(semid,0,0);
  val = semctl(semid,0,GETVAL);
  if (val==-1) perror("Lettura valore semaforo");
  printf("Valore semaforo %d: %d\n",semid,val);
  n= fork();
  if (n == -1) perror("Fork");

  if (n == 0)
  {
    sleep(3);
    printf("Do via libera\n");
    up(semid,0); 
  }
  else
  {
    down(semid,0); 
    printf("Ricevuto via libera\n");
    sleep(3);
    if (semctl(semid,0,IPC_RMID)==-1) perror("Rimozione semaforo");
  }
}

In generale prima della rimozione bisognerebbe attendere la terminazione degli altri processi che possono operare sul semaforo; in questo caso particolare la segnalazione up(semid,0) da parte del figlio serve in pratica anche come segnalazione della sua terminazione.
Si noti che (scorrettamente) non si va a verificare il risultato di up e down (che per come sono definite è il risultato delle system calls semop chiamate al loro interno). Utilizzi "seri" di questa system call dovrebbero tener conto, in particolare, del fatto che una semop sospensiva può terminare non perché è stata fatta una up, ma perché il processo ha ricevuto un segnale, o il semaforo è stato rimosso (vedere come al solito il man e i testi di riferimento).

Nella descrizione della system call semop non si fa menzione sull'ordine in cui diversi processi sospesi su uno stesso semaforo vengono risvegliati. Il seguente esempio può servire a verificare, almeno per curiosità, quale sia il meccanismo implementato in una specifica realizzazione della chiamata di sistema. Infatti vengono generati, a distanza di 1 secondo l'uno dall'altro, 5 processi che vanno a fare down su un semaforo "rosso". Poi il genitore effettua, sempre a distanza di un 1 secondo l'una dall'altra, 5 chiamate di up.
Anche se non si devono fare assunzioni sulla velocità dei processi, in un sistema scarico (senza altri processi sottoposti dall'utente) il tempo di 1 secondo tra un evento e l'altro è enormemente maggiore del tempo reale necessario per far arrivare i processi alle chiamate di down e quindi si può effettivamente supporre che queste chiamate avvengano nell'ordine, prima da parte del processo con i=0, poi da parte di quello con i=1, eccetera. Le printf (a patto che non interferisca la bufferizzazione dell'output) possono fornire una conferma, e in ogni caso volendo essere più sicuri si può aumentare il parametro passato a sleep, o sostituire la chiamata con il for (di opportuna lunghezza a seconda della velocità della CPU) messo come commento.
In ogni caso, anche se verifichiamo che l'implementazione di semop che stiamo utilizzando sembra risvegliare i processi nell'ordine in cui sono stati sospesi, non si potrebbe concludere che sia sempre così, e in ogni caso non bisogna scrivere programmi che si basino su questa ipotesi, perché in una diversa versione del sistema potrebbero non funzionare. Se ci interessa assicurare che lo scheduling di diversi processi in attesa di un evento segua una certa politica (FIFO o altre), bisogna utilizzare i cosiddetti semafori privati , come visto nel corso.

**************************************************************
USO DI SEMOP PER OPERAZIONI PIU GENERALI DI DOWN E UP
**************************************************************

Un primo tipo di operazione più generale delle down e up "classiche" che si può realizzare con semop consiste nell'uso nel campo sem_op di valori positivi e negativi con valore assoluto diverso da 1.
Questo può servire con un semaforo "a contatore", inizializzato ad un numero N di "gettoni" (es. corrispondente ad un numero di risorse uniformi di cui richiedere l'utilizzo in mutua esclusione); una operazione in cui il campo sem_op ha un valore negativo "-k" corrisponde al tentativo di prendere in un solo colpo k gettoni (risorse), infatti comporta il tentativo di decrementare di k il valore del semaforo, che tuttavia non viene effettuato se il valore risultante del semaforo dovesse risultare negativo; in questo caso il processo viene sospeso.
Si noti che questa operazione è diversa da una sequenza di k operazioni down, perché nel caso il valore s del semaforo sia strettamente compreso fra 0 e k (es. k=5 e s=2), con le k operazioni down in sequenza, le prime s hanno successo (le prime 2 nell'esempio) decrementando il valore del semaforo fino a zero, e il processo si sospende sulla successiva (la terza nell'esempio); invece con una singola operazione con sem_op=k, il valore del semaforo rimane invariato, lasciando la possibilità ad un altro processo di effettuare operazioni con sem_op negativo e |sem_op| <= s.

Una operazione con sem_op=k (positivo) corrisponde al rilascio di k gettoni (risorse).
Nel caso in cui vi siano diversi processi sospesi in attesa di completare una operazione con sem_op negativo, e il valore del semaforo venga incrementato (non necessariamente di 1), viene portato a termine (risvegliando i processi corrispondenti) un insieme di operazioni completando le quali il valore del semaforo rimane non negativo.
Più esattamente, un insieme "massimale" di operazioni che abbia la proprietà precedente, cioè un insieme di operazioni tale che le operazioni in esso possono essere tutte completate, ma una volta completate quelle, nessuna delle operazioni rimaste in sospeso può essere completata senza rendere il semaforo negativo.
Ad esempio, se il valore del semaforo è 0, e vi sono 3 processi p1, p2 e p3, sospesi in attesa di completare operazioni sul semaforo con valore di sem_op di -2, -4 e -2 rispettivamente, se il semaforo viene incrementato di 4 (operazione con sem_op=4) potrebbero essere risvegliati alternativamente:
  • p1 e p3; oppure:
  • p2
indipendentemente dall'ordine in cui i processi sono stati sospesi. Una specifica implementazione probabilmente scorrerà la lista di processi sospesi sul semaforo (lista in cui può darsi che i processi siano stati deliberatamente tenuti nell'ordine in cui si sono sospesi) effettuando per ciascuno l'operazione sospesa, se ora è possibile completarla. Una tale implementazione nell'esempio risveglierebbe:
  • p2 solo se è il primo dei 3 ad essere preso in considerazione;
  • p1 e p3 altrimenti

**************************************************************

Un secondo tipo di operazione diversa da down e up riguarda l'attesa del valore zero.
Come si è visto una chiamata di semop con valore di sem_op = 0 fa sospendere i processi se il semaforo non vale zero (fino a quando il valore diventa zero); questo è l'uso opposto al "normale", non va assolutamente mescolato con operazioni la cui logica segue quella di down e up (eventualmente generalizzate come già visto).
Può essere utile per realizzare un meccanismo chiamato "barriera" (es. sul testo di Tanenbaum), in cui abbiamo N processi il cui codice prevede una prima sezione (es. di inizializzazione) e una seconda sezione (in cui ad esempio i processi devono interagire per una elaborazione comune), e vogliamo che tutte le prime sezioni siano eseguite prima di tutte le seconde sezioni, cioè che i processi arrivati al termine della prima sezione proseguano solo dopo che anche gli altri vi sono arrivati.
Questo tipo di sincronizzazione può essere realizzato sfruttando il caso sem_op=0 come nel seguente esempio, inizializzando il semaforo a N e facendo chiamare ad ogni processo al termine della prima sezione una down (che però non è usata con la logica delle down e up classiche) seguita da una operazione (che chiameremo "waitz") di attesa del valore zero.
Le varie down hanno in questo caso solo l'effetto di decrementare il valore del semaforo e non sono mai sospensive; quando l'ultimo processo P arriva alla down, il valore diventa zero, gli altri processi in attesa sulla "waitz" vengono sbloccati, e P stesso, alla successiva chiamata di waitz, non viene sospeso ma prosegue. In questo modo tutti i processi accedono alla seconda sezione.
Le sleep(i) eseguite da ogni processo fanno in modo da distanziare il tempo reale di arrivo al termine della prima sezione da parte dei diversi processi.
::::::::::::::
semz.c
::::::::::::::
#include "semfun.h"

#include <stdio.h>
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/sem.h>
#include <sys/wait.h>
#include <unistd.h>
#include <stdlib.h>

#define N 5

int semid;

int waitz(int semid, int semnum)
        /* attesa del valore zero sulla componente semnum di semid */
{
  struct sembuf sb;

  sb.sem_num=semnum;
  sb.sem_op=0;
  sb.sem_flg=0;
  return semop(semid,&sb,1);
}

void proc(int i)
{ 
  sleep(i);
  printf("Processo %d con pid %d al termine della prima sezione\n",i,getpid());
  down(semid,0);
  waitz(semid,0);
  printf("Processo %d con pid %d all'inizio della seconda sezione\n",i,getpid());
}

int main()
{
  int val,pid,i;

  semid = semget(IPC_PRIVATE,1,0600);
  printf("Valore semid: %d\n",semid);
  if (semid == -1) perror("Creazione semaforo");
  seminit(semid,0,N);
  val = semctl(semid,0,GETVAL);
  if (val==-1) perror("Lettura valore semaforo");
  printf("Valore semaforo %d: %d\n",semid,val);

  for(i=0;i<N;i++)
        if (fork()==0)
                { proc(i); exit(0);};
  for(i=0;i<N;i++)
        { pid=wait(0);
          printf("Terminato processo %d\n",pid);
        }
  if (semctl(semid,0,IPC_RMID)==-1) perror("Rimozione semaforo");
}

**************************************************************

Un terzo senso in cui semop è più generale di down e up consiste nel fatto, che finora abbiamo soltanto menzionato, che a semop si può passare un vettore di operazioni su elementi (tipicamente diversi) dello stesso vettore di semafori.
In questo caso, come si può leggere nel man, semop effettua le operazioni in modo "atomico" cioè indivisibile. Vale a dire, se almeno una delle operazioni non può essere portata a termine (es. operazione con campo sem_op negativo, con valore assoluto maggiore di quello del semaforo), il processo (a meno che non venga richiesto diversamente attraverso i flag delle operazioni) viene sospeso e non viene effettuata nessuna delle operazioni descritte nel vettore passato a semop. Le operazioni verranno effettuate tutte insieme soltanto in un momento successivo, quando i valori dei semafori saranno tali da rendere possibili tutte le operazioni.


******************************
ESEMPIO: I 5 FILOSOFI
******************************

Il seguente esempio realizza la simulazione del problema dei "5 filosofi" soggetta a deadlock, in quanto vengono utilizzati 5 semafori per l'allocazione in mutua esclusione delle risorse (forchette) e il processo (filosofo) i acquisisce le 2 risorse con due operazioni down sui semafori i e (i+1)%5.

Il fatto che la soluzione sia soggetta a deadlock non significa che questo effettivamente si verifichi, l'evento dipende anche dallo scheduling della CPU (che pur non essendo affatto programmato per evitare il deadlock, determina l'effettivo avanzamento dei processi), e dagli altri eventi che occorrono nel sistema operativo nel momento in cui il programma gira.

Questo esempio utilizza la memoria condivisa, ma solo per tenere traccia dello stato dei filosofi per visualizzarlo all'utente; tale stato non è usato esplicitamente per concedere o meno l'allocazione delle risorse, che dipende invece dallo stato dei semafori: un processo ottiene l'allocazione delle forchette superando le due down.
::::::::::::::
filodead.c
::::::::::::::
#include "semfun.h"

#include <stdio.h>
#include <stdlib.h>
#include <signal.h>
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/sem.h>
#include <sys/shm.h>
#include <sys/wait.h>
#include <unistd.h>
#include <stdlib.h>

/* 5 filosofi con possibile deadlock */

int semid,shmid;
char *stato; /* puntera' ad un array di 5 caratteri in 
	        memoria condivisa che serve solo per illustrare
		 all'utente lo stato dei 5 filosofi, 
		T=thinking, H=hungry, E=eating */

void cleanup(int sig)
{
  if (semctl(semid,0,IPC_RMID) == -1)
     perror("semctl");
  if (shmctl(shmid,IPC_RMID,0) == -1)
     perror("semctl");
  exit(sig);
}

void print()	/* stampa lo stato dei filosofi */ 
{
  int i;
  for(i=0;i<5;i++) putchar(stato[i]);
  printf("\n");
}

void proc(int i)
{
  int j,n;
  int m;

  m = 50000000 + 30000000 * i; /* tempo per cui usera' le forchette; 
	i filosofi con i piu' alto sono piu' lenti a mangiare, questo differenzia
	un pochino il loro comportamento e su qualche sistema favorisce 
	il verificarsi del deadlock */

  for (n=0;n<50;n++)
  {
    stato[i]='H';
    printf("FILOSOFO %d PRENDE LA %d  ",i,i);
    print();
    down(semid,i);
    down(semid,(i+1)%5);
    stato[i]='E';
    printf("FILOSOFO %d MANGIA       ",i);
    print();
    for(j=0;j<m;j++); /* ogni processo usa le risorse per un tempo diverso */
    stato[i]='T';
    printf("FILOSOFO %d SMETTE       ",i);
    print();
    up(semid,(i+1)%5);
    up(semid,i);
  }
}

int main()
{
  int i,j;
  pid_t pid;
  struct sigaction sa;

  if ((semid = semget(IPC_PRIVATE,5,0600))==-1)
     perror("semget");
  if ((shmid = shmget(IPC_PRIVATE,5,0600))==-1)
     perror("shmget");
  stato = (char *) shmat(shmid,0,0);
  if ((int)stato == -1) perror("shmat");
  for (i=0;i<5;i++)
  {
    stato[i]='T';
    seminit(semid,i,1);
  }

  for(i=0;i<5;i++)
	{
	if (fork()==0)
		{ proc(i); exit(0);}
	}

  sa.sa_handler = cleanup;
  sigaction(SIGINT,&sa,NULL);
     /* se vanno in deadlock, bisogna interrompere tutto,
	 il processo padre associa cleanup a SIGINT per rimuovere 
	semafori e memoria condivisa in questo caso */

  for(i=0;i<5;i++)
	{ 
	  pid=wait(0);
	  printf("Terminato processo %d\n",pid);
	}
  cleanup(0);

}

ESERCIZIO 6.3: sfruttando il fatto che a semop si può passare un vettore di operazioni su elementi dello stesso vettore di semafori, operazioni che vengono effettuate in modo indivisibile, modificare la soluzione precedente in modo che non sia soggetta a deadlock.

************************************************
ESEMPIO: PRODUTTORI E CONSUMATORI
************************************************

Questo esempio utilizza in modo più essenziale la memoria condivisa per far condividere un "buffer" di caratteri a processi "produttori" e "consumatori" che, rispettivamente, depositano e prelevano i caratteri.
Nell'esempio specifico, il programma principale fa partire un certo numero di processi produttori, che producono ripetutamente sempre uno stesso carattere (diverso a seconda del processo), e un unico processo consumatore, che estrae i caratteri dal buffer e li scrive su un file, il cui nome deve essere passato come argomento al comando (es. generare l'eseguibile "prodcons" e chiamare "prodcons pippo"). Sullo standard output viene stampato (per avere una traccia) il contenuto del buffer dopo ogni operazione.
In questo caso particolare, dopo che i processi produttori sono terminati, il processo padre scrive un carattere convenzionale nel buffer che viene interpretato dal processo consumatore come "fine dati".

::::::::::::::
prodcons.c
::::::::::::::

/********  N produttori, 1 consumatore con processi e semafori ********/

#include "semfun.h"

#include <stdio.h>
#include <signal.h>
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/sem.h>
#include <sys/shm.h>
#include <sys/wait.h>
#include <unistd.h>
#include <stdlib.h>

#define N 5    		 /* dim. buffer */
#define NPROD 5    	/* numero produttori */
#define NGIRI 20 	/* quanti caratteri prodotti da ogni produttore */

struct buffer{
        char pool[N];
        int in,out,count;
        } *buf;		/* puntatore al buffer che viene allocato 
			   in memoria condivisa */

int shmid,mutex,empty,full;

void printstatus()	/* stampa contenuto del buffer */
{
  int i;
 
  printf("Contenuto del buffer:");
  for (i=0;i<buf->count;i++)
      putchar(buf->pool[(buf->out+i)%N]);
}

void put(char i)
{
 
  buf->pool[buf->in]=i;
  buf->in = (buf->in+1)%N;
  buf->count++;
 
  printstatus();
  printf(" dopo put del carattere %c\n",i);
  fflush(stdout); /* cosi' va bene anche se si ridirige l'output su disco */
}

void get(char *ip)
{

  *ip=buf->pool[buf->out];
  buf->out = (buf->out+1)%N;
  buf->count--;
  
  printstatus();
  printf(" dopo get del carattere %c\n",*ip);
  fflush(stdout); /* cosi' va bene anche se si ridirige l'output su disco */
}
 

void producer(char c)	/* produce NGIRI volte il carattere c */
{
  int i;
 
  for (i=0;i<NGIRI;i++)
        {
 
	int j;
        for (j=0;j<10000000;j++); /* per rallentare */

	down(empty,0);
        down(mutex,0);
        put(c);
        up(mutex,0);
	up(full,0);
        }
}

void consumer(char *filename)	/* copia da buffer a filename */
{
  char c;
  FILE *res;
 
  res=fopen(filename,"w");

  do /* esce quando trova il carattere \0 nel buffer */
  {
  	int i;
 
  	for (i=0;i<10000000;i++); /* per rallentare */
 
  	down(full,0);
  	down(mutex,0);
  	 get(&c);
  	up(mutex,0);
  	 up(empty,0);
	
  	putc(c,res);
  }
  while (c!=0);
}
 

void clear(int s)	/* rimuove strutture IPC */
{

if (semctl(mutex,0,IPC_RMID) == -1) perror("semctl");
if (semctl(full,0,IPC_RMID) == -1) perror("semctl");
if (semctl(empty,0,IPC_RMID) == -1) perror("semctl");
if (shmctl(shmid,IPC_RMID,0) == -1) perror("shmctl");
exit(s);
}
 
int main(int argc, char *argv[])
{

int i,pid;
struct sigaction sa;

if (argc != 2) 
{
	fprintf(stderr,"un nome di file come argomento\n");
	 exit(1);
}

if ((mutex = semget(IPC_PRIVATE,1,0666))==-1)
     perror("semget");
seminit(mutex,0,1);

if ((empty = semget(IPC_PRIVATE,1,0666))==-1)
     perror("semget");
seminit(empty,0,N);
 
if ((full = semget(IPC_PRIVATE,1,0666))==-1)
     perror("semget");
seminit(full,0,0);
 
 
/* acquisisce memoria condivisa per il buffer */
 
if ((shmid = shmget(IPC_PRIVATE,sizeof(struct buffer),0666))==-1)
     perror("shmget");
buf = (struct buffer *) shmat(shmid,0,0);
if ((int) buf == -1) perror("shmat");
 
 sa.sa_handler = clear;
sigaction(SIGINT,&sa,NULL);

/* inizializza buffer */
 
buf->in = buf->out = buf->count = 0;
 
/* genera produttori e consumatore */

 
fflush(stdout);


for(i=0;i<NPROD;i++)
	{
	if (fork()==0)
		{ producer('0'+i); exit(0);}
	}
if (fork()==0)
	{ consumer(argv[1]); exit(0);}


for(i=0;i<NPROD;i++)
	{ pid=wait(NULL);
	  printf("Terminato processo %d\n",pid);
	}

/* produce carattere 0 per far terminare il consumatore */
down(empty,0);
down(mutex,0);
put(0);
up(mutex,0);
up(full,0);

wait(NULL);
clear(0);
}
