**********************************

IL MANUALE ON-LINE DI UNIX

**********************************

Il comando:
	man nome
fornisce  il manuale on-line per la voce (comando, system call, funzione di libreria o altro) di nome "nome", es.:
	man ls
In caso di ambiguità (comandi, funzioni e system calls con lo stesso nome):
	man -s section nome
Le sezioni più rilevanti sono:
  1. comandi
  2. system calls
  3. funzioni di libreria (con sottosezioni)
Se come "nome" si usa "intro" si ha un manuale introduttivo per la sezione.

Per le system calls e le funzioni di libreria, man informa su quale file bisogna includere per poter usare la funzione, in particolare per far vedere al compilatore la dichiarazione del prototipo della funzione (il che, qualora sia necessario rammentarlo, serve a verificare che la funzione venga chiamata con parametri corretti in numero e tipo) e delle eventuali costanti simboliche utilizzate dalla funzione come parametri di ingresso o uscita.

**********************************

COMPILATORE E LINKER

**********************************

In questa sezione richiamiamo in modo semplificato alcuni concetti relativi alla compilazione e collegamento (linking) con compilatori come cc o gcc; per i concetti di base vedasi ad esempio il testo dei corsi di architettura degli elaboratori, "Structured Computer Organization", di A.S. Tanenbaum (sezione 7.4 della 4a edizione).

La produzione di un file eseguibile a partire da un sorgente prevede in generale due fasi:

1) la compilazione, cioè traduzione del linguaggio sorgente (il C in questo caso) in linguaggio oggetto, in questo caso linguaggio macchina per l'architettura considerata, es. x86 o Sparc.
In questa fase molti nomi simbolici (di variabili e funzioni) vengono sostituiti da indirizzi, nel caso il compilatore veda la definizione delle variabili (cioè la dichiarazione che alloca la memoria per le variabili, vedasi la distinzione tra "dichiarazione" e "definizione" in C) o delle funzioni (cioè la dichiarazione con il corpo della funzione, non solo il prototipo) e quindi il compilatore traduca anche il "corpo" della funzione, assegnandogli fra l'altro un indirizzo della prima istruzione, quella a cui saltare quando la funzione viene chiamata.
Se però il compilatore non ha questà possibilità, rimangono dei nomi simbolici dove le variabili vengono usate, o le funzioni vengono chiamate. Questi nomi vengono collezionati in una tabella dei simboli esterni che viene inclusa nello stesso file del codice oggetto.

2) il "collegamento" (linking), che consiste nell'eliminare i nomi simbolici rimasti e creare un unico file eseguibile, a partire da uno o più file oggetto.

Il collegamento serve per uno dei seguenti motivi, o entrambi:

a) Perché il programmatore, per questioni di modularità, divide il sorgente in più file e li compila separatamente in diversi file oggetto.
Tuttavia, trattandosi di file con cui si vuole ottenere un unico eseguibile, in almeno uno si chiameranno funzioni definite in un altro, allora è appunto necessario il "collegamento" che ad esempio nella chiamata di una funzione "f" (il cui codice é in un file oggetto diverso da quello dove c'è la chiamata) sostituisce il nome simbolico "f" con l'indirizzo della prima istruzione di f.
Questo indirizzo è quello che risulta nel file eseguibile, che contiene almeno il codice di tutte le funzioni definite dall'utente, e quindi è (semplificando) la concatenazione del codice contenuto nei diversi file oggetto, con queste successive modifiche.

b) Perché il programmatore include chiamate di funzioni facenti parte di librerie predefinite, cioè funzioni che vengono chiamate nel programma senza fornirne la definizione; anche per queste funzioni deve essere effettuato il collegamento al loro codice, e ci sono due modi per farlo:
  • statico, cioè effettuato al momento della generazione del file eseguibile;  in analogia al caso precedente, questo meccanismo include nel file eseguibile il codice delle funzioni di libreria, ma ha due difetti: moltiplica copie del codice delle funzioni nei diversi file eseguibili che le utilizzano, e rende necessaria la ricompilazione nel caso di aggiornamenti della libreria;
  • dinamico, cioè effettuato (dopo opportuna preparazione al momento della generazione del file eseguibile) solo quando il file viene caricato in memoria per essere eseguito; questo meccanismo, sui cui dettagli di realizzazione qui non entriamo, elimina i difetti precedenti, permettendo di mantenere un'unica copia (sia su file che in memoria) delle funzioni di libreria che vengono quindi condivise da parte di più programmi; questo è il meccanismo largamente usato sia in Unix che in Windows (es. la più o meno nota estensione .dll in Windows sta per Dynamic Link Library)
Si noti che anche nel caso "a" il collegamento potrebbe essere dinamico ma se ne ottiene un vantaggio soltanto se il programmatore riutilizza in diverse applicazioni uno stesso pacchetto di funzioni da lui realizzate, o ne aggiorna periodicamente la realizzazione. Se il programmatore mette queste funzioni a disposizione di altri, si scivola nel caso "b".

Il comando (cc o gcc) con cui si chiama il compilatore C è molto versatile e permette di chiamare implicitamente anche o solo il linker (collegatore).
I diversi modi per chiamare il compilatore riguardo alle due fasi (compilazione vera e propria e collegamento) sono:

1) Compilare e collegare uno o più files (costituenti un unico programma) con un solo comando, es.:
	gcc file1.c file2.c file3.c
per default il nome dell'eseguibile generato e' "a.out", se se ne vuole uno diverso si aggiunge 
	-o nomeFileEseguibile
Se nei sorgenti viene richamata una funzione di una libreria, in generale si aggiungerà l'opzione "-l...." dove al posto dei
puntini viene specificata la libreria, es. "-lpthread" per la libreria dei Posix threads che verrà utilizzata più avanti nel corso. In alcuni casi, come la libreria standard del C - quella di printf, getchar, etc - e la libreria delle chiamate di sistema che verrano utilizzate nel resto del corso, non è necessario indicare esplicitamente il nome della libreria.

2) Compilare e collegare in due fasi (per agevolare questo si usa l'utility make descritta più avanti).
La sola compilazione avviene con l'opzione "-c" che genera un file .o, esempio:
	gcc -c file1.c
genera "file1.o". Questo lo si fa per tutti i file sorgente (uno o più) che costituiscono il programma.
Qui non è necessario menzionare librerie dato che non si tratta di fare collegamenti.

Il solo collegamento avviene chiamando ad esempio:
	gcc file1.o file2.o file3.o
dove è necessario passare tutti i file oggetto che devono costituire il programma, ed usare l'opzione per le tutte le librerie usate nei diversi file.
Al solito, l'eseguibile viene chiamato "a.out" o con il nome passato con l'opzione "-o"

**********************************

IL COMANDO MAKE

**********************************

Il comando make è una utility per semplificare la compilazione di programmi costituiti da moduli separati.

In un file, tipicamente con nome "makefile", si tiene traccia delle dipendenze dei files; questo permette di rigenerare un file eseguibile solo se qualche file da cui dipende è stato modificato, o meglio, ha data o ora più recente (questa è di solito una buona approssimazione del fatto che il file sia stato modificato dopo la compilazione).

Ad esempio, consideriamo un programma costituito da due moduli, il cui sorgente è nei file "main.c" e "mod1.c". Un makefile opportuno può contenere quanto segue: 
program: main.o mod1.o
    gcc -o program main.o mod1.o
main.o: main.c
    gcc -c main.c
mod1.o: mod1.c
    gcc -c mod1.c
questo significa che "program" dipende da "main.o" e da "mod1.o"; quando uno dei due è più recente di program, o quest'ultimo non esiste, si "ricostruisce" program eseguendo l'azione associata dopo un <tab> nella riga successiva (è possibile usare più righe di azioni).
Questo viene fatto dopo aver "ricostruito" (se è il caso) i file da cui dipende.

Per chi vuole saperne di più, le parti per ricostruire main.o e mod1.o potrebbero essere omesse; per ricostruire un file che non ha azioni vengono usate regole di default, per default "file.o" dipende da "file.c" e l'azione da eseguire è "cc -c file.c", o "gcc -c file.c" se make è configurato opportunamente per modificare il nome di default del compilatore C.
Non possono però essere omesse del tutto nel caso in cui i due moduli includano un file "defs.h" con definizioni di costanti e prototipi delle funzioni definite in un file e chiamate nell'altro, bisogna dare esplicitamente la dipendenza: 

program: main.o mod1.o
    gcc -o program main.o mod1.o
main.o: main.c defs.h
    gcc -c main.c
mod1.o: mod1.c defs.h
    gcc -c mod1.c
(solo i comandi gcc -c potrebbero ancora essere omessi).

Se il file che descrive dipendenze e azioni si chiama "makefile" basta chiamare il comando "make" senza argomenti, altrimenti:
	make -f nome-del-makefile
Se non viene specificato come argomento un file da ricostruire (detto "target"), si usa il primo che si trova nel file.

Il fatto che il "target" venga ricostruito quando non esiste un file con lo stesso nome ha una conseguenza importante: se ho un'applicazione costituita da diversi programmi (che non "dipendono" uno dall'altro, nel senso di make), o comunque voglio comandare con un solo makefile e un solo breve comando la ricompilazione di diversi programmi (ad esempio
una serie di programmi tutti nella stessa directory), si può usare un "target" fittizio (nel senso che non esisterà mai un file con quel nome).

Esempio: 

all: program1 program2 program3
program1: main1.o mod1-1.o
    gcc -o program1 main1.o mod1-1.o
program2: ...
program3: ...
Il target "all" non ha azioni associate, serve solo per far ricostruire program1, program2, program3 chiamando solo "make" o "make all" invece di "make program1 program2 program3"; questo perchè make vede che non esiste un file di nome "all" e quindi procede ad esaminare se deve ricostruire "program1", poi "program2", poi "program3"; infine, andrebbe ad eseguire l'azione associata ad "all", ma non ve ne è nessuna, in particolare nessuna che genera un file di nome "all", quindi alla prossima chiamata di "make" il trucco funzionerà di nuovo.

Per imparare a usare make si possono provare le seguenti varianti: 
	make -n
non esegue le azioni, dice solo quali sono le azioni che verrebbero eseguite. 
	make -d
scrive i motivi per cui fa (o farebbe, se si usa -n) le azioni. 
	touch nomefile
modifica la data e ora di "nomefile" con quelle correnti, senza modificare il contenuto (un modo veloce per vedere le azioni che make fa, o farebbe).

**************************************

ALCUNE SYSTEM CALLS PER GESTIONE FILES

**************************************

Forniscono I/O a basso livello. Le vediamo come primi esempi perché si tratta di operazioni (aprire un file, leggere o scrivere su file) il cui significato è noto anche a chi non conosce ancora molto dei sistemi operativi.
É bene sapere che esistono, ma tipicamente per operare su files è meglio usare le funzioni di libreria, che sono implementate con le chiamate di sistema ma offrono diversi vantaggi (si vedranno più avanti alcune differenze).
Le chiamate di sistema operano sui files tramite "descrittori di file": numeri interi usati come indice nella tabella dei file aperti. I descrittori 0, 1, 2 corrispondono ai file che costituiscono i cosiddetti "stream" per lo standard input, standard output e standard error di un programma.
Lo standard input, per default associato alla tastiera (quando la finestra selezionata è quella da cui è stato fatto partire il programma), è il file da cui leggono funzioni come getchar e scanf, mentre lo standard output, per default associato alla finestra in cui il programma è stato lanciato, è quello su cui scrivono funzioni come putchar e printf.
Lo stream per lo "standard error" esiste per dare la possibilità di separare l'output "normale" di un programma e i messaggi di errore.

Le seguenti sono descrizioni delle chiamate di sistema molto semplificate rispetto a quelle che si trovano nel "man":
OPEN:
	int open(const char *path, int  oflag,  /*  mode_t  mode  */ ...);
Apre il file avente pathname "path" con modalità di apertura "oflag"; restituisce un descrittore di file.
Con il bit opportuno settato in oflag crea il file, se non esiste, con i diritti di accesso passati in "mode". 

CREAT:
	int creat(const char *path, mode_t mode);
Crea il file avente pathname "path" e diritti di accesso "mode". 

READ:
	ssize_t read(int fildes, void *buf, size_t nbyte);
Legge "nbyte" bytes dal file associato a "fildes" (un descrittore restituito da una open precedente) copiandoli nella zona di memoria a partire dall'indirizzo "buf"; restituisce il numero di bytes effettivamente letti 

WRITE:
	ssize_t write(int fildes, const void *buf, size_t nbyte);
Scrive "nbyte" bytes nel file associato a "fildes" copiandoli dalla zona di memoria a partire dall'indirizzo "buf"; restituisce il numero di bytes effettivamente scritti. 

CLOSE:
	int close(int fildes);
Chiude il file associato a "fildes". 

LSEEK:
	off_t lseek(int fildes, off_t offset, int whence);
sposta il puntatore di lettura/scrittura di "fildes" del numero di bytes "offset" a partire dalla posizione specificata da "whence" (dall'inizio, dalla posizione corrente o dalla fine). Il valore precedente del puntatore di lettura/scrittura identifica, come numero di byte, il punto del file dove è terminata l'ultima operazione (di lettura o scrittura) sul file aperto, o meglio l'ultima effettuata attraverso l'apertura identificata da "fildes" (su un file vi possono infatti essere più aperture contemporaneamente, ognuna ha un puntatore diverso). Ciò naturalmente a patto che l'ultima operazione sia stata read o write e non lseek. 

DUP e DUP2:
	int dup(int fildes);
	int dup2(int fildes, int fildes2);
dup restituisce un descrittore (il più piccolo libero) che viene associato allo stesso file di "fildes", cioè copia la riga della tabella dei file aperti identificata da "fildes" nella prima riga libera della tabella stessa;
dup2 fa in modo che fildes2 sia associato allo stesso file di fildes, copiando la riga della tabella dei file aperti identificata da "fildes" in quella identificata da "fildes2".
Sono utilizzate per la ridirezione dei file standard, ad esempio dall'interprete dei comandi per interpretare correttamente comandi come "ls > pippo", in cui si vuole che l'output del comando venga diretto sul file "pippo".

ESEMPI

**********************************

Il seguente programma è un esempio di uso di system calls su files e di gestione di errori di sistema: 
:::::::::::::: 
readerr.c 
::::::::::::::

#include <stdio.h>
#include <fcntl.h>
main()
{
  int f,i,n,ne;
  char buf[20];

  f=open("fileprova",O_RDONLY);
  if (f==-1) syserr("Errore apertura file");
  else
	{
	printf("Numero bytes da leggere (max 20): ");
	scanf("%d",&n);
	if (n>20) n=20;
	ne=read(f,buf,n);
	printf("letti %d caratteri: ",ne);
	for (i=0;i<ne;i++) putchar(buf[i]);
	printf("\n");
	}
}
::::::::::::::
syserr.c
::::::::::::::
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <errno.h>

void syserr(char *msg)
{


fprintf(stderr,"ERROR: %s (%d",msg,errno);
fprintf(stderr,"; %s)\n",strerror(errno));
exit(1);

}

::::::::::::::
Il secondo file contiene la definizione di una procedura per la stampa di messaggi di errore che potrebbe essere utilizzata in tutti i programmi che usano le system calls; conviene tuttavia utilizzare l'analoga funzione "perror" che fa parte della libreria standard.
Tutte le system calls (che restituiscono un intero) in caso di errore restituiscono il valore -1 e valorizzano la variabile intera "errno" con un valore diverso per ogni tipo di errore.
Il programma nell'esempio utilizza la funzione strerror per ottenere un sintetico messaggio di errore appropriato per ogni tipo di errore.
La variabile "errno" non deve essere dichiarata, ma è necessario includere il file <errno.h> che conterrà una opportuna definizione, che è piuttosto complicata per motivi che qui non possiamo approfondire.
Si noti che i messaggi di errore sono stampati sullo stream standard error con fprintf(stderr, ...) invece che sullo standard output con printf. É bene seguire sempre questa convenzione che dà la possibilità di ridirigere separatamente l'output "normale" di un programma (con "> nomefile") e i messaggi di errore (con "2>  nomefile" nella bash e in alcune altre shell, ">& nomefile" in altre ancora).
Il file errno.h, da includere con la notazione <errno.h>  (che come noto indica che il file da includere si trova in una opportuna directory, tipicamente  /usr/include/, dove vengono messi tutti i file .h delle librerie standard), definisce nomi di costanti per i diversi possibili codici di errore; a questi nomi simbolici si fa riferimento nel man della chiamata di sistema, nella sezione "ERRORS". Ad esempio per open "ENOENT" (che presumibilmente sta per "error: no entry") indica che il file da aprire non esiste (e non è stato chiesto di crearlo con opportune opzioni di open).

ESERCIZIO 1.3: trovare questa informazione nel "man".

I valori possibili di errno e i nomi di costanti possono essere utilizzati per programmare opportunamente il comportamento in caso di errore, specie nel caso si voglia fare qualcosa di diverso dalla segnalazione di un messaggio di errore, o lo si voglia personalizzare:
  if (f==-1) if (errno == ENOENT) ....

Un modo corretto di compilare il programma è chiamare:

Ma in generale per compilare programmi suddivisi, per motivi di buona organizzazione del codice, in diversi files (es. avere una sola versione della funzione syserr in un apposito file), è utile usare i makefile e il comando make.

ESERCIZIO 1.5: eseguire il programma (contenuto nel file readerr se lo si è compilato con make):  
  •    in presenza e in assenza di un file di nome "fileprova";
  •    ridirigendo lo standard output e lo standard error.
Si noti che anche per la "read" sarebbe necessario verificare se ci sono errori.
Per non appesantire troppo la scrittura dei programmi che utilizzano le chiamate di sistema, ma d'altra parte non tralasciare la verifica degli errori, si può effettuare quest'ultima in modo uniforme per tutte le chiamate di sistema definendo una funzione per effettuare le chiamate di sistema.
I seguenti files call1.h e call1.c costituiscono un microscopico esempio di definizione di una libreria, in cui definiamo delle funzioni (una sola in questo caso) che possono essere riutilizzate in tanti altri programmi. In questo caso costerebbe poco copiare e incollare la definizione della funzione "call" in tutti i file che la usano e ricompilarla tutte le volte, ma usare file separati facilita la modifica della libreria.
Si noti che esiste un file "header" (intestazione) che contiene solo la dichiarazione della funzione call (il suo prototipo, o interfaccia: quanti parametri ha e di che tipo, qual è il tipo restituito) e un file .c che ne contiene la definizione.
Il file .c che usa la funzione include il file "call1.h" in modo che il compilatore possa "vedere" la dichiarazione della funzione che viene chiamata in esso, e nella generazione del file eseguibile deve essere effettuato il collegamento con il codice della funzione call.
Si noti che la definizione della funzione è piuttosto semplice in quanto essa prende semplicemente il risultato della chiamata di sistema, se si è verificato un errore (il risultato è -1) chiama  la "perror" passandogli il messaggio aggiuntivo da stampare passato alla Call stessa, e termina l'esecuzione del programma; cioè qualunque errore che occorre in una chiamata di sistema viene considerato "fatale" per il proseguimento del programma, con la chiamata di exit che forza la terminazione del programma. Per convenzione, un valore diverso da 0 (in questo caso, 1) passato ad exit indica la terminazione a causa di un errore. Si vedrà in seguito chi può utilizzare questa informazione.

::::::::::::::
call.h
::::::::::::::
#include <stdlib.h>

int call(int res, char * msg);
::::::::::::::
call.c
::::::::::::::
#include "call.h"

int call(int res, char * msg)
{
	if (res == -1)
	{
		perror(msg);
		exit(1);
	}
	else return res;
}
::::::::::::::
readerr1.c
::::::::::::::
#include <stdio.h>
#include <fcntl.h>
#include "call.h"

main()
{
  int f,i,n,ne;
  char buf[20];

  call(f=open("fileprova",O_RDONLY),"Errore apertura file");
  printf("Numero bytes da leggere (max 20): ");
  scanf("%d",&n);
  call(ne=read(f,buf,n),"Errore Read");
  printf("letti %d caratteri: ",ne);
  for (i=0;i<ne;i++) putchar(buf[i]);
  printf("\n");
}
::::::::::::::

Una diversa realizzazione potrebbe lasciare a call solo il compito di stampare il messaggio di errore, se necessario, e lasciare al programma chiamante il compito di decidere se proseguire o no.
In questo caso va utilizzato il valore restitiuito dalla funzione chiamando ad esempio:
if ((call(open("fileprova",O_RDONLY),"Errore apertura file")) == -1 ) ....

RIDIREZIONE 

**********************************

Esempio di ridirezione dello standard output: 

::::::::::::::
redir.c
::::::::::::::
#include <stdio.h>

main()
{  int n;
   n=creat("pippo",0600);
   close(1); /* chiude lo standard output */
   dup(n);   /* ora il descrittore 1 e' associato a pippo */
   close(n);
   printf("Hello \n");  /* oppure write(1,"Hello \n",7); */
}
Al posto di 
close(1);
dup(n);
si può usare:
dup2(n,1)
In entrambi i casi la stampa del messaggio, che è effettuata con printf, cioè sullo standard output, ha luogo sul file "pippo".

DIFFERENZA TRA L'USO DI SYSTEM CALLS E FUNZIONI DI LIBRERIA PER L'I/O

**********************************

Le funzioni della libreria di I/O standard del linguaggio C sono realizzate mediante chiamate di sistema ed effettuano una "bufferizzazione" dei dati a livello del programma utente; cioè, ad esempio, per la scrittura, usano un buffer (letteralmente "tampone", nell'informatica: una zona di memoria utilizzata per parcheggiare temporaneamente dei dati) in cui vengono accumulati i byte che il processo scrive con printf, putchar e simili; solo in certi momenti il buffer viene svuotato sul file effettuando una chiamata di sistema write:
  • quando è pieno;
  • per file associati a un terminale (non un disco), quando si ha una riga intera (cioè quando si scrive un "\n");
  • quando il file viene chiuso (con fclose);
  • quando il processo (programma) termina di sua iniziativa, cioè arriva al termine del suo "main", oppure chiama la funzione "exit";
  • quando viene chiamata la funzione: 
		int fflush(FILE *stream);
In lettura (es. chiamata di getchar o scanf), viene letto un blocco di dati nel buffer, se questo è vuoto; altrimenti si leggono i dati dal buffer precedentemente riempito.

Per saperne di più consultare ad es. il classico testo sul C di Kernighan & Ritchie, cap.8, e il manuale on line.

Il tipo di bufferizzazione di default dipende dal tipo di dispositivo a cui il file è associato; per file su disco il default è a blocchi, ad es. di 1024 bytes, o 2048, etc; per i terminali, a righe.
Se si vuole cambiarla per un file si può usare la funzione setvbuf (vedere man setvbuf) una volta per tutte subito dopo aver aperto il file. Se si vuole alterare il meccanismo solo in opportuni momenti, usare fflush.

In generale è conveniente usare le funzioni di libreria e non le system calls, perché offrono molte funzionalità in più (es. printf formatta l'output in vari modi, per esempio permette di scrivere valori numerici in formato decimale) e perché si fanno meno chiamate al sistema operativo, che comportano un certo overhead (lavoro di gestione aggiuntivo) dovuto al passaggio alla modalità kernel. Ad esempio è pesante chiamare molte volte read o write per un carattere alla volta, mentre se ad es. si effettuano ripetute chiamate di putchar, solo ogni tanto (quando c'è da svuotare il buffer, quindi una volta ogni 1024, o 2048, etc) queste chiamate daranno luogo a una chiamata di sistema.

Lo si può notare con i due programmi seguenti.
::::::::::::::
loopsc.c
::::::::::::::
#include <stdio.h>

main(int argc, char *argv[])
{
    int n=1000,i;

    if (argc==2) n=atoi(argv[1]);

    for(i=0;i<n;i++)
        write(1,"Q",1);
}
    
::::::::::::::
looplf.c
::::::::::::::
#include <stdio.h>

main(int argc, char *argv[])
{
    int n=1000,i;

    if (argc==2) n=atoi(argv[1]);

    for(i=0;i<n;i++)
        putchar('q');
}

Entrambi scrivono ripetutamente un carattere, il primo con la chiamata di sistema, il secondo con la funzione di libreria, per un numero di volte passato come argomento.

ESERCIZIO 1.7. Dopo aver generato gli eseguibili "loopsc" e "looplf", chiamare ad esempio:

    time ./looplf 1000 > /dev/null

    time ./loopsc 1000 > /dev/null

proseguendo con valori più grandi dell'argomento (10000, 100000, 1000000,...) si noterà una differenza sempre più significativa tra i tempi di esecuzione delle due versioni. Il comando "time" esegue il comando sul resto della riga, fornendo il tempo "real" trascorso in totale, il tempo "user" di esecuzione in user mode, il tempo "sys" di esecuzione in kernel mode. Per evitare di generare file enormi, l'output viene ridiretto su /dev/null, un "finto" dispositivo che non fa nulla dei byte che riceve. 

In casi molto particolari ci sono anche delle differenze di risultato, come nei seguenti due programmi, che scrivono alcuni bytes e poi effettuano un ciclo infinito: 
::::::::::::::
loop.c
::::::::::::::
main()
{  
   write(1,"Hello \n",7); 
   for (;;);
}
::::::::::::::
loop1.c
::::::::::::::
#include<stdio.h>
main()
{  
   printf("Hello \n"); 
   for (;;); 
}
Essi sembrano equivalenti: entrambi scrivono e poi devono essere interrotti.
Se però si ridirige l'output dei due programmi su un file su disco, oppure non si stampa il \n, sono ancora equivalenti? Si noti che quando il programma (processo) termina perché viene interrotto, il buffer non viene svuotato.

ESERCIZIO 1.8: usare fflush o setvbuf per rendere il loro comportamento analogo anche in questi casi.
